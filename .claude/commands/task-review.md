---
description: 指定タスクIDのコミット差分をレビューして指摘を報告する
argument-hint: <タスクID 例: T1-3>
---

タスクID `$ARGUMENTS` の実装コミットをレビューしてください。**これはレビューのみのコマンドです。コードの修正・phase文書のチェックボックス操作は一切行わないこと。**

## 1. 対象コミット収集

- `git log --grep "Task: $ARGUMENTS" --format="%h %s"` で該当コミット群を取得する（`/impl` が全実装コミットに `Task: <タスクID>` トレーラーを付けている）
- 誤ヒット防止のため、各コミットの本文に正確なトレーラー行 `Task: $ARGUMENTS` があることを確認する（例: `T1-1` の検索が `T1-10` に誤ヒットしないこと）
- 0件の場合は「該当コミットが見つからない」旨を報告して停止する

## 2. 差分取得

- 各コミットを `git show <hash>` で確認する
- 複数コミットある場合は、最古のコミットの親から最新コミットまでの範囲diff（`git diff <oldest>^..<newest>`）でタスク全体の変更も俯瞰する

## 3. コンテキスト読込

- `docs/plan/phase-<N>-*.md` の該当タスクブロック（説明・完了条件・実施履歴）
- 完了条件が参照している `docs/spec/` の受入基準（AC）
- `docs/implementation-rules.md`（規約の判定基準として全章）

## 4. レビュー観点

`docs/implementation-rules.md` §7（Definition of Done）と §6 に基づき、以下を確認する:

1. **完了条件・AC充足**: タスクの完了条件と参照specのACを差分が満たしているか。仕様にない実装が紛れ込んでいないか
2. **テスト**: AC由来テストが `[AC-n]` 命名で存在するか（§5の配置・書き方規約に従っているか）
3. **検証コマンド**: `npm run check` と `npx astro build` を実際に実行して通るか（`package.json` 未作成等で実行不能な場合は省略し、その旨をレビュー結果に明記する）
4. **ドキュメント反映**: 仕様変更・知見が該当文書に反映されているか。SSoT原則に反する重複記載がないか
5. **コミット規約**: メッセージが Conventional Commits + 日本語サブジェクト + `Task:` トレーラー（§6）に従っているか
6. **コード品質**: §2〜§4の規約違反（named export・命名・`any`・言語ポリシー等）、バグ、エッジケースの見落とし

## 5. 結果のファイル出力

レビュー結果を `docs/archive/review/$ARGUMENTS.md` に書き出す（ディレクトリがなければ作成する）。

- 記載内容:
  - レビュー日
  - 対象コミット一覧（ハッシュ + サブジェクト）
  - DoDチェック結果（観点1〜6それぞれの合否と根拠）
  - 指摘一覧（重要度順: **要修正** / **推奨** / **軽微**。各指摘に該当ファイル・行と理由を付ける）
  - 総合判定（承認 / 要修正）
- 同名ファイルが既に存在する場合（再レビュー）は、既存内容を消さず末尾に `---` 区切りで新しいレビュー結果を追記し、履歴を残す
- 出力した結果のファイルはGitコミットすること

## 6. 報告

チャットにも総合判定と指摘のサマリーを報告する。要修正の指摘がある場合は、修正後に再度 `/task-review $ARGUMENTS` を実行するよう案内する。
