---
description: 指定タスクIDの実装を仕様駆動で行い、完了時にコミットする
argument-hint: <タスクID 例: T1-3>
---

タスクID `$ARGUMENTS` の実装を、以下の手順で行ってください。

## 1. タスク特定

- タスクIDは `T<フェーズ番号>-<連番>` 形式（例: `T1-3`）。形式が不正なら理由を報告して停止する
- フェーズ番号に対応する `docs/plan/phase-<N>-*.md` を開き、該当タスクのブロック（チェックボックス + 説明 + 完了条件）を探す
- タスクが見つからない場合、または既に `- [x]`（完了済み）の場合は、その旨を報告して停止する

## 2. Planモード確認

- 現在Planモードでない場合はユーザーにその旨を警告し、続行するか確認する

## 3. コンテキスト読込

実装を始める前に必ず読むこと:

- 該当タスクの説明本文と**完了条件**（多くは spec の AC 番号を参照している）
- 完了条件が参照している `docs/spec/` の該当spec（特に受入基準の章）
- `docs/implementation-rules.md`（全章。特に §2〜§5 の言語・コーディング・テスト規約）
- 必要に応じて `docs/architecture.md`・`docs/markdown-pipeline/`（Sätteriプラグイン関連タスクの場合は必読）

## 4. モデルタグ確認

タスクに `〔Fable 5〕` タグが付いている場合、現在のモデルがFable 5でなければユーザーにその旨を警告し、続行するか確認する（モデル運用ルールは `docs/plan/README.md` を参照）。

## 5. 実装

- **仕様にない振る舞いを実装しない**。仕様の変更・詳細化が必要になったら、**先に該当specを更新**してから実装する（仕様駆動）
- implementation-rules の規約に従う: 識別子は英語・コメント/テスト名は日本語（§2）、named exportのみ・kebab-case・`any`禁止（§3）、テストは `tests/` ミラー配置で AC由来テストは `[AC-n]` 命名（§5）
- 実装中に得た知見（落とし穴・制約）は該当リファレンス文書（architecture / markdown-pipeline）へ反映する
- 必要に応じて「astro-docs」や「context7」のmcpサーバーを利用して最新の情報を取得するように

## 6. 検証

- `npm run check` と `npx astro build` を実行し、両方通ることを確認する
- `package.json` が未作成の初期タスク（T1-1以前の状態）では実行できないため省略し、省略した旨を最終報告に明記する

## 7. 進行状況更新

- 完了条件をすべて満たしたら、phase文書の該当チェックボックスを `- [ ]` → `- [x]` に更新する
- フェーズ内の全タスクが完了し「フェーズ完了条件」を満たした場合は、`docs/plan/README.md` のフェーズ一覧の状態列も更新する

## 8. コミット（必須）

実装完了時は必ずGitへコミットする。

- リポジトリがgit未初期化の場合: git初期化がタスク自体に含まれるなら（T1-1）先に初期化する。含まれないのに未初期化なら異常なので報告して停止する
- コミットメッセージ規約（`docs/implementation-rules.md` §6）:
  - Conventional Commits + 日本語サブジェクト（type: `feat` / `fix` / `docs` / `test` / `refactor` / `chore`）
  - **末尾にトレーラー `Task: <タスクID>` を必ず付与する**（例: `Task: T1-3`）。レビューコマンドがこのトレーラーで対象コミットを収集する
- 1タスク内で意味のある単位ごとに複数コミットしてよいが、**すべてのコミットにトレーラーを付ける**
- コミット前に `npm run check` を通す（実行可能な状態であれば）

## 9. 実施内容を報告

- フェーズ番号に対応する `docs/plan/phase-<N>-*.md`の`## 実施履歴`の`### <タスクID>`にて今回対応した内容の概要と特記事項があれば記載する
- SSoTの原則に則り、別ドキュメントに記載した内容は該当箇所へのリンクを記載し、重複した内容を記載しないようにすること
- 概要には以下を含めること
  - 実装内容のサマリー（変更ファイル・満たした完了条件/AC）
  - 検証結果（check / build の結果。省略した場合はその理由）
  - 作成したコミットの一覧（ハッシュ + サブジェクト）
