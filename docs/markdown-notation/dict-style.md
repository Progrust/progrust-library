# 辞書エントリのスタイルガイド

辞書コンテンツ（`content/dict/**/*.md`）を執筆する際の規約。記法そのものの一覧は [`rule.md`](rule.md)、frontmatterの項目定義は [`frontmatter.md`](frontmatter.md) を参照（本書には重複記載しない）。

## 対象と前提

- 辞書は「短い単一トピック」のコンテンツ（content-model R-1）
- 記事・本の中の `[[wikilink]]` をホバーすると、プレビュー小窓に**辞書の本文全文**が表示される（wikilink-ui R-7）。そのため各エントリは**短く自己完結**していることが大原則

## ファイル名（slug）の命名

ファイル名（拡張子を除く）がそのままslugになり、`[[slug]]`・URL `/dict/[slug]` として使われる（content-model R-4/R-5）。

- **英語の概念名をkebab-case ASCII**で付ける（例: `ownership`, `borrow-checker`, `trait-object`）
- 悪い例: `Rust所有権.md`（非ASCII）、`traitObject.md`（camelCase）、`the-ownership.md`（冗長な冠詞）
- slugは `content/dict/` 全体で一意（フォルダが違っても重複不可。content-model R-6）。**作成前に必ず既存ファイル一覧を確認**し、同一概念のエントリや命名の揺れ（`mutability` と `mutable` 等）を作らないこと
- `content/dict/` 配下のディレクトリ構成は**整理用として自由**に切ってよい（例: `rust-language/01-basic/`）。slug・URLはファイル名だけで決まり、配置場所の影響を受けない（content-model R-5）。新規エントリは関連エントリと同じディレクトリに置く

## エントリのテンプレート

基本は以下の固定構成で書く。
ただし、以下の構成で満足な説明ができない場合は適宜見出しなどを追加して拡張可能。

1. **概要** — 冒頭1〜2段落。見出しは付けず、最初の段落だけ読めば概要が掴める文にする。長文にはしない。
2. **コード例** — 最小限のコード1個（見出しは付けず定義の真下に記載する）。ただし、辞書の内容によってコード記載は不要と判断した場合は省略可。
3. **詳細** — 概要だけでは語り切れない場合はここで見出しを設けて詳細を記載する。複数見出し可能。（不要なら省略可）
4. **補足** — 発展的内容・例外・歴史的経緯などは `:::details` で折りたたむ（`## 補足` 見出し。不要なら省略可）

- 見出しレベル1（`#`）は使わない（タイトルはfrontmatterの `title`）。本文の見出しは `##` から
- 区切り線（`---`）は使わない

雛形（コピーして使う）:

````markdown
---
title: 配列型（Array Type）
description: 同じ型の要素を固定長で並べたプリミティブ型。要素数はコンパイル時に確定する型の一部。
created_at: 2026-07-18
updated_at: 2026-07-18
tags: ["型システム", "基本文法", "プリミティブ型", "複合型"]
public: false
---

配列型`[T; N]`は、同じ型の要素を固定長で並べた[[primitive-type]]です。要素数`N`はコンパイル時に確定する[[constant]]で、型自体の一部になります。そのため`[i32; 3]`と`[i32; 4]`は異なる型として扱われます。

```rust playground
fn main() {
    let scores: [i32; 3] = [80, 95, 70]; // 3科目の点数
    println!("2科目目: {}", scores[1]);
    println!("要素数: {}", scores.len());
}
```

概要では語り切れない内容がある場合はここに記載する。

## 補足

:::details[同じ値で埋めるリピート式]
折りたたまれる補足をここに書く。
:::

````

## 記法の活用

テンプレートの構成を守ることを第一としたうえで、文章だけで説明するより分かりやすくなる場合は、[`rule.md`](rule.md) にある記法を**積極的に**使う。目安:

- 比較・対応関係・選択肢の整理 → **テーブル**（例: `const` と `static` の違い）
- 注意・警告・豆知識など本文と区別したい短い言及 → **メッセージ**（`:::message{warning}` / `{tip}` 等。種別は内容に合わせる）
- 処理の流れ・状態遷移・構造の関係 → **mermaid**（フローチャート・シーケンス図等）
- 用語の出典・厳密な定義の参照 → **脚注**・引用

ただし装飾のための乱用はしない。プレビューには本文全文が表示されるため、大きくなりがちな要素（mermaid・長いテーブル）は本文の分量とのバランスを見て `:::details` 内に置くことも検討する。判断基準は「その記法にすることで読者の理解が速くなるか」。

## 文体と分量

- 本文は**ですます調**で書く
- frontmatterの `description` は**体言止め**で簡潔に（SEOディスクリプション用）
- 分量の目安: 折りたたみ（`:::details` 内）を除く本文200〜600字＋コード例1個。プレビュー小窓で読み切れる長さに保つ
- ただしこれは**あくまで目安**であり、目安を守るために必要な内容を削ってはならない。トピックの理解に必要なら超過してよい（超過分はまず `:::details` への移動を検討する）
- 執筆を依頼した人が記載内容を具体的に指示した場合、その内容を分量目安のために圧縮・省略しない

## コード例の規則

- Rustコードは `rust playground` 指定を基本とし（記法は [`rule.md` のコードブロック節](rule.md#コードブロック)）、**Rust Playground（stable, edition 2024）でそのまま動く**ように書く
- 題材は日本語話者に身近なものを選ぶ（買い物・残高などの生活例。文化依存の強い英語圏の慣用例は避ける）。日本語はコメント・文字列リテラル・出力メッセージで使う
- 識別子（変数名・関数名・型名）は**英語**で付ける。**ローマ字表記の日本語は禁止**（例: `zandaka` ✗ → `balance` ✓、`kosuu` ✗ → `count` ✓）。実際のRustコードの慣習に合わせるため
- **あえてコンパイルエラーになる例**は、期待するエラーコードをコード内コメントで明記する（例: `// エラー: E0502（可変借用と不変借用の共存）`）
- コードは検証スクリプト `npm run check:dict` でコンパイル検証される。検証の挙動をマーカー（コードブロックの**直前の行**に置くHTMLコメント）で制御できる:
  - マーカーなし → コンパイル成功を要求（`fn main` がないブロックは自動で `fn main() {}` にラップされる）
  - `<!-- rustc: expect E0502 -->` → コンパイルが**失敗**し、かつ指定エラーコードが出ることを要求
  - `<!-- rustc: skip -->` → 検証対象外（擬似コード・断片など）。乱用しない

## 辞書リンク（wikilink）の方針

記法は [`rule.md` の辞書リンク節](rule.md#辞書リンク) を参照。

- 本文中に既存の辞書エントリに相当する語が出たら、**初出の1回を必ず** `[[slug]]` でリンクする（任意ではなく必須。2回目以降は張らない）
- wikilinkの表示テキストはリンク先の `title` になる（wikilink-ui R-1。例: `[[variable]]` → 「変数（Variable）」）。文中に埋め込むと表示が変わるため、リンク挿入後の文が自然に読めるよう周辺の表現を調整する
- `## 関連項目` には**本文中でリンクしなかった**関連エントリのみを列挙する（本文でリンク済みの項目は重複させない。関連はあるが本文に登場しない概念の置き場）
- リンクを張る前に対象slugのファイルが `content/dict/` に**実在することを確認**する（存在しないslugへのリンクはビルドエラー。content-model R-13）
- リンクしたい対象がまだ存在しない場合はリンクせず、`<!-- TODO: [[slug]] 作成後にリンク -->` のHTMLコメントを残す
- TODOコメントに書くslugは**将来の正式slugの予約**として扱う。仮の名前ではなく、上記の命名規則（英語概念名・kebab-case）に従って本決定のつもりで付け、既存のTODOコメントに同じ概念の予約があればその表記に必ず揃える（後日エントリを作成する際は、この予約slugがそのままファイル名になる）
- frontmatter内ではwikilinkは使えない（[`frontmatter.md` の補足](frontmatter.md#frontmatterに辞書linkは記載できない)）

## タグの付け方

- 使用中のタグは `npm run dict:tags` で集計する（手動の一覧表は持たない。この出力を正とする）
- タグは既存タグから選ぶことを優先する（1エントリ1〜3個目安）。既存タグと意味が重なる近縁の新タグを乱造しない（表記ゆれ例: `ownership` ではなく既存の `所有権` を使う）
- 適切な既存タグがない場合のみ新タグを付けてよい。新タグを使ったときは最終報告・コミットで明示する

## 日付と公開フロー

- `created_at` / `updated_at` は実際の当日日付（`date +%F` で取得した値）を設定する。新規作成時は両方、既存エントリの更新時は `updated_at` のみ更新する
- 新規エントリは `public: false`（下書き）で作成する。`public: true` への切り替え（公開）は人間がレビューのうえで行う
- 既存エントリを更新する場合は全面書き換えせず差分修正に留め、既存の文体・構成を保持する
