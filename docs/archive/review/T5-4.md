## 指摘一覧

### 要修正

1. **spec R-2「一覧ページの絞込でも同じインデックスを利用する」と実装方式が食い違ったまま** — `docs/spec/search.md:5`（§1 概要）および `docs/spec/search.md:17`（R-2）
   spec は一覧絞込も検索インデックス JSON を利用すると2箇所で明記しているが、実装（`src/scripts/list-filter.ts`）はインデックスを fetch せず、カードの `data-title` / `data-description` / `data-tags` 属性から読む方式。architecture §8 は以前から `data-*` 参照方式を規定しており、文書の優先順位（spec > architecture）に照らすと spec との矛盾が未解消。DOM 方式自体は合理的（fetch 不要・R-11 を構造的に充足・マッチ判定は `entryMatches` 共有でセマンティクス同一）なので、**実装を変えるのではなく spec §1 / R-2 の当該記述を実態（絞込データはカードの data 属性から取得し、マッチ判定ロジックを検索ボックスと共有する）に合わせて更新し、決定を plan 実施履歴に記録する**ことを求める。CLAUDE.md の「仕様の変更が必要になったら先に該当 spec を更新」の原則に関わるため要修正とする。

### 推奨

1. **キーワード入力の aria-label に「（準備中）」が残っている** — `src/components/ListFilter.astro:45`
   `initListFilter()` が `disabled` を除去して機能を有効化した後も、スクリーンリーダーには `aria-label="一覧内をキーワードで絞り込み（準備中）"` が読み上げられる。T5-3 では同じ理由で `SearchBox.astro` の aria-label から「（準備中）」を除去しており、それと不整合。除去して JS 無効時の案内が必要なら placeholder 等で対応するのが一貫する。

### 軽微

1. **`entryMatches` 呼び出しでダミーの `type`/`url` を合成している** — `src/scripts/list-filter.ts:36-38`
   `entryMatches` が `SearchEntry` 全体を要求するため `{ ...card, type: "dict", url: "" }` を合成している。`entryMatches` の第1引数を `Pick<SearchEntry, "title" | "description" | "tags">` にすれば合成が不要になり、判定が使うフィールドも型で明示できる。挙動に影響はないため現状可。
2. **R-11 括弧書き「ただしタグ詳細ページは全種別」の読みが曖昧** — `docs/spec/search.md:35`
   タグ詳細ページにも絞込 UI があり対象が全種別、とも読める文言だが、実装ではタグ詳細ページは絞込 UI 対象外（T5-4 実施履歴に設計判断として記録済み）。spec 側の文言を「タグ詳細ページには絞込 UI を置かない（一覧自体が全種別）」等に明確化すると将来の誤読を防げる。要修正1の spec 更新と同時に直すとよい。

## 総合判定

**要修正**。コード・テスト・検証はいずれも良好で AC-7 の振る舞いは満たしているが、spec search.md §1 / R-2 の「一覧絞込もインデックス JSON を利用する」という記述と実装方式（`data-*` 属性参照）の矛盾が未解消のまま残っており、仕様駆動・SSoT の原則上ここで止める。修正は spec の文言更新（+ plan 履歴への決定記録）のみでよく、コード変更は不要。修正後に再度 `/task-review T5-4` を実行すること。あわせて推奨1（aria-label の「（準備中）」除去）の対応を勧める。
