# docs ディレクトリの構成とドキュメント管理ルール

## 構成（地図）

| ディレクトリ | 読者 | 内容 |
| --- | --- | --- |
| [`spec/`](spec/README.md) | 実装者 | 機能別の詳細仕様書（要求仕様 + 受入基準。仕様駆動開発の起点） |
| [`architecture.md`](architecture.md) | 実装者 | 全体設計（プロジェクト構成・コレクション定義・モジュール分割・データフロー） |
| [`implementation-rules.md`](implementation-rules.md) | 実装者 | 横断的な実装ルール（ツールチェーン・言語ポリシー・コーディング規約・テストの書き方・Git運用・完了チェックリスト） |
| [`plan/`](plan/README.md) | 実装者 | フェーズ分割した実装計画（1タスク=1セッション・完了条件付き。全フェーズ完了で凍結→archive） |
| [`markdown-notation/`](markdown-notation/rule.md) | 執筆者 | 本サイトで利用可能なMarkdown記法の一覧（書き方のみ。実装の話は書かない） |
| [`markdown-pipeline/`](markdown-pipeline/README.md) | 実装者 | Sätteriカスタムプラグインの機能別実装リファレンス（実装方式・雛形コード・落とし穴・残課題） |
| `ui-design/` | 実装者 | UIデザインの仕様・計画（+ モック用ダミーコンテンツ）。モックHTML本体はリポジトリ外・claude design 上（[`ui-design/ui-design-spec.md`](ui-design/ui-design-spec.md) 冒頭にリンク） |
| `archive/` | — | クローズ済み文書の凍結アーカイブ（歴史的記録。更新しない） |

リポジトリ直下の `CLAUDE.md` はアプリ概要と本ディレクトリへの地図（詳細仕様は持たない）。

## ドキュメント作成・管理ルール

新しくドキュメントを作成・更新する際は、以下のルールに従うこと。

1. **1つの事実は1つの文書にのみ書く**
   同じ情報を複数の文書に書かない。他の文書から必要な場合は相対リンクで参照する（更新時のズレを防ぐ）。
2. **読者で文書を分ける**
   執筆者向け（記法・書き方）と実装者向け（実装方式・コード・落とし穴）を同じ文書に混在させない。
3. **CLAUDE.mdは地図に徹する**
   CLAUDE.mdには仕様上の決定事項（各1〜3行）と参照リンクのみを置き、実装詳細・長い注意書きは書かない。詳細は`docs/`配下の該当文書に置く。
4. **検証・計画系の文書はクローズしたら凍結してアーカイブする**
   クローズ時に冒頭へ凍結注記（クローズ日・「更新しない」・現行の参照先）を付けて `docs/archive/` へ移動する。そこにある生きた知識（実装方式・雛形コード・落とし穴）は、凍結前にリファレンス文書へ抽出しておく。
5. **実装リファレンスは統一構成で書く**
   `markdown-pipeline/` の各文書は「記法 → 実装方式 → 雛形コード → 落とし穴と回避策 → 制約・残課題」の構成に揃える。「たぶん動く」は書かず、動作確認した事実と未検証の残課題を明確に区別する。
6. **R/AC番号は文書名付きで参照する**
   specのR/AC番号は文書ごとの独立採番で衝突するため、他文書から参照するときは必ず文書名を付ける（`pages R-18`・`wikilink-ui R-17` 形式）。**自文書内の自己参照のみ裸番号可**（spec文書内の「タイブレークはR-4に従う」等）。コード・テストでの参照規約は [`implementation-rules.md`](implementation-rules.md) 3章・5章。
